# 跨平台集成测试计划

## 概述

本文档描述了减肥对赌 APP 的跨平台集成测试计划,确保 Android、iOS 和后端 API 之间的数据互通和功能一致性。

## 测试环境

### 后端环境
- **API 服务器**: 本地开发服务器或测试环境
- **数据库**: PostgreSQL (测试数据库)
- **缓存**: Redis (测试实例)
- **支付网关**: Stripe 测试模式

### 移动端环境
- **Android**: 
  - 模拟器: Android 13 (API 33)
  - 真机: Android 10+ 设备
- **iOS**: 
  - 模拟器: iOS 16+
  - 真机: iOS 14+ 设备

## 测试场景

### 场景 1: 跨平台用户注册和登录

**目标**: 验证用户可以在一个平台注册,在另一个平台登录

**测试步骤**:
1. 在 Android 上注册新用户
2. 验证后端创建了用户记录
3. 在 iOS 上使用相同凭证登录
4. 验证用户信息一致
5. 反向测试 (iOS 注册 → Android 登录)

**预期结果**:
- ✅ 用户可以在任意平台注册
- ✅ 用户可以在任意平台登录
- ✅ 用户信息在两个平台上一致
- ✅ JWT 令牌在两个平台上都有效

### 场景 2: 跨平台创建和接受对赌计划

**目标**: 验证在一个平台创建的计划可以在另一个平台查看和接受

**测试步骤**:
1. 用户 A 在 Android 上创建对赌计划
   - 设置赌金: 100 元
   - 设置时长: 30 天
   - 设置目标: 减重 5kg
2. 验证后端创建了计划记录
3. 用户 A 在 Android 上邀请用户 B
4. 用户 B 在 iOS 上查看计划详情
5. 用户 B 在 iOS 上接受计划
6. 验证计划状态变为 active
7. 在 Android 和 iOS 上都能看到计划已激活

**预期结果**:
- ✅ Android 创建的计划在 iOS 上可见
- ✅ 计划详情在两个平台上一致
- ✅ iOS 可以接受 Android 创建的计划
- ✅ 资金冻结正确执行
- ✅ 双方都收到通知

### 场景 3: 跨平台打卡和进度同步

**目标**: 验证在一个平台打卡后,另一个平台能看到更新

**测试步骤**:
1. 用户 A 在 iOS 上打卡
   - 输入体重: 78.5kg
   - 上传照片
   - 添加备注
2. 验证后端保存了打卡记录
3. 用户 A 在 Android 上查看打卡历史
4. 验证打卡记录显示正确
5. 用户 B 在 Android 上查看用户 A 的进度
6. 验证进度统计正确
7. 用户 B 在 iOS 上打卡
8. 用户 A 在 Android 上查看用户 B 的进度

**预期结果**:
- ✅ iOS 打卡在 Android 上可见
- ✅ Android 打卡在 iOS 上可见
- ✅ 打卡数据完整 (体重、照片、备注)
- ✅ 进度统计在两个平台上一致
- ✅ 体重变化图表正确显示

### 场景 4: 跨平台结算流程

**目标**: 验证结算在两个平台上都能正确显示

**测试步骤**:
1. 等待计划到期或手动触发结算
2. 验证后端执行结算逻辑
3. 在 Android 上查看结算结果
4. 在 iOS 上查看结算结果
5. 验证结算金额分配正确
6. 验证余额更新正确
7. 验证双方都收到结算通知

**预期结果**:
- ✅ 结算自动触发
- ✅ 结算结果在两个平台上一致
- ✅ 资金转移正确执行
- ✅ 余额在两个平台上同步
- ✅ 通知在两个平台上都收到

### 场景 5: 跨平台支付流程

**目标**: 验证支付操作在两个平台上都能正确执行和同步

**测试步骤**:
1. 用户 A 在 Android 上充值 500 元
2. 验证后端更新余额
3. 在 iOS 上查看余额
4. 验证余额显示正确
5. 用户 A 在 iOS 上提现 200 元
6. 在 Android 上查看余额
7. 验证余额更新正确
8. 在两个平台上查看交易历史

**预期结果**:
- ✅ 充值在两个平台上都能看到
- ✅ 提现在两个平台上都能看到
- ✅ 余额在两个平台上同步
- ✅ 交易历史在两个平台上一致
- ✅ 冻结余额正确显示

### 场景 6: 跨平台通知功能

**目标**: 验证通知在两个平台上都能正确接收

**测试步骤**:
1. 用户 A 在 Android 上创建计划并邀请用户 B
2. 验证用户 B 在 iOS 上收到邀请通知
3. 用户 B 在 iOS 上接受计划
4. 验证用户 A 在 Android 上收到生效通知
5. 测试打卡提醒通知
6. 测试结算通知

**预期结果**:
- ✅ FCM 通知在 Android 上正常接收
- ✅ APNs 通知在 iOS 上正常接收
- ✅ 通知内容正确
- ✅ 通知点击跳转正确
- ✅ 通知权限正确处理

### 场景 7: 跨平台社交功能

**目标**: 验证社交功能在两个平台上的一致性

**测试步骤**:
1. 用户 A 在 Android 上发表评论
2. 用户 B 在 iOS 上查看评论
3. 用户 B 在 iOS 上回复评论
4. 用户 A 在 Android 上查看回复
5. 在两个平台上查看排行榜
6. 验证排行榜数据一致
7. 在两个平台上查看勋章

**预期结果**:
- ✅ 评论在两个平台上同步
- ✅ 排行榜在两个平台上一致
- ✅ 勋章在两个平台上一致
- ✅ 鼓励功能正常工作

### 场景 8: 离线同步测试

**目标**: 验证离线数据在网络恢复后正确同步

**测试步骤**:
1. 用户 A 在 Android 上断开网络
2. 用户 A 创建离线打卡记录
3. 验证数据保存到本地队列
4. 恢复网络连接
5. 验证数据自动同步到服务器
6. 在 iOS 上查看同步后的数据
7. 测试冲突处理 (同一天在两个平台都打卡)

**预期结果**:
- ✅ 离线数据保存到本地
- ✅ 网络恢复后自动同步
- ✅ 同步后数据在另一平台可见
- ✅ 冲突正确处理 (服务器数据优先)

### 测试环境配置

#### 后端配置
```bash
# 启动测试数据库
docker-compose -f docker-compose.test.yml up -d

# 运行数据库迁移
python manage.py migrate

# 启动测试服务器
python manage.py runserver 0.0.0.0:8000
```

#### Android 配置
```bash
# 设置测试 API 地址
# 在 app/src/debug/res/values/config.xml 中设置
<string name="api_base_url">http://10.0.2.2:8000</string>

# 运行 Android 应用
./gradlew installDebug
```

#### iOS 配置
```bash
# 设置测试 API 地址
# 在 Config/Debug.xcconfig 中设置
API_BASE_URL = http://localhost:8000

# 运行 iOS 应用
xcodebuild -workspace WeightLossBetting.xcworkspace \
  -scheme WeightLossBetting \
  -destination 'platform=iOS Simulator,name=iPhone 14' \
  build
```

### 常见问题排查

#### 问题 1: 数据不同步
- 检查网络连接
- 检查 API 响应
- 检查本地缓存
- 检查同步逻辑

#### 问题 2: 通知未收到
- 检查通知权限
- 检查设备令牌注册
- 检查 FCM/APNs 配置
- 检查后端通知发送日志

#### 问题 3: 支付失败
- 检查 Stripe 配置
- 检查测试卡号
- 检查支付回调
- 检查余额更新逻辑
